# Question 5 — Custom Retractions on the Sphere

## Goal

Compare three retractions on the sphere for the SteepestDescent optimizer:

1. **Default:** $R_v(x) = \frac{x + v}{\|x + v\|}$ (normalization)
2. **New (Eq. 4):** $R_v^{\mathrm{new}}(x) = x\,\frac{4 - \|v\|}{4 + \|v\|} + v\,\frac{4}{4 + \|v\|}$ (Cayley-type)
3. **Wrong:** $R_v^{\mathrm{wrong}}(x) = x + v$ (no projection back to sphere)

## Implementation

Created two subclasses of `Sphere` that only override the `retraction` method, inheriting all other geometric tools (tangent space projection, inner product, norm, random point, etc.). This is possible since pymanopt's `Sphere` class has a clean OOP design.

## Experiments

- Channel: Red, $k = 30$, SteepestDescent, 100 iterations.
- Recorded per-iteration cost and Riemannian gradient norm via `log_verbosity=2`.
- Verified whether the final point satisfies $\|M^*\|_F = 1$ (i.e., stays on the sphere).

## Expected Observations

### Default vs New retraction
Both are **valid retractions** on the sphere — they satisfy $R_0(x) = x$ and $\frac{d}{dt}R_{tv}(x)\big|_{t=0} = v$. Both should converge, potentially at slightly different rates due to different local approximation quality.

### Wrong retraction
$R_v^{\mathrm{wrong}}(x) = x + v$ does **not** map back to the sphere. After the first step, the iterate leaves the manifold ($\|x + v\| \neq 1$ in general). This means:
- The "Riemannian gradient" computed by pymanopt is no longer meaningful (it projects onto the tangent space of a point not on the manifold).
- The algorithm does **not** solve the constrained problem — it degenerates into unconstrained gradient descent in $\mathbb{R}^{(n-1) \times k}$.
- The cost may still decrease, but the solution is not on the sphere and therefore does not correspond to a valid Laplacian.
